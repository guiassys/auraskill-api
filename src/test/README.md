# 🧪 Test Development Guidelines

This document establishes the mandatory architectural steps and operational procedures for engineering new automated tests within this project.

---

## 🛠️ Prerequisites & Environment Verification

Before developing or executing tests, ensure your local subsystem meets the following operational state:
1.  **Docker Daemon Active:** Docker must be running with WSL 2 integration explicitly enabled for the active Linux distribution.
2.  **Profile Isolation:** All test executions must target the isolated environment defined in `src/test/resources/application-test.yml`. Do not hardcode connection strings or credentials.

---

## 🏗️ Step-by-Step Test Creation Workflow

To introduce a new test suite or scenario into the testing ecosystem, you must complete the following sequential phases:

### Phase 1: Feature Specification (Behavior Layer)
1.  Navigate to the `src/test/resources/features/` directory.
2.  Create a new file with the `.feature` extension matching the context of the business domain being targeted.
3.  Write the behavioral scenarios using formal Gherkin syntax rules (`Feature`, `Scenario`, `Given`, `When`, `Then`, `And`).

### Phase 2: Context Binding (Glue Layer)
1.  Navigate to the designated test steps package under `src/test/java/`.
2.  Create a dedicated step definition Java class.
3.  Map every Gherkin text expression directly to an executable Java method using Cucumber step annotations.

### Phase 3: Context Configuration & Infrastructure Activation
1.  Decorate the test suite entry-point class or execution runner with the `@ActiveProfiles("test")` annotation.
    *   Note: This is mandatory to prevent the application from loading default environment contexts and to trigger the automatic Testcontainers lifecycles.
2.  Ensure that the test context inherits or initializes the appropriate Spring Boot testing utilities (`@SpringBootTest`, `TestRestTemplate`, or `MockMvc`) to perform integration routines against the ephemeral container database.

## ⚠️ Critical Development Rules (Security & Database)

When creating step definition classes (Glue Layer) that perform HTTP requests, you **must** follow these two architectural rules to avoid `401 Unauthorized` and `OptimisticLocking` errors:

### 1. Mandatory HTTP Authentication
The API enforces JWT/Keycloak security. All HTTP requests made via `TestRestTemplate` must explicitly include a bearer token to trigger the `TestSecurityConfig` mock decoder.
*   **Action:** Create a helper method in your steps class to append the mock token and use `restTemplate.exchange` instead of standard naked methods (like `postForEntity`).

```java
private HttpHeaders getAuthenticatedHeaders() {
HttpHeaders headers = new HttpHeaders();
headers.set("Authorization", "Bearer mocked-token");
return headers;
}

// Example usage inside a step:
HttpEntity<MyRequest> entity = new HttpEntity<>(request, getAuthenticatedHeaders());
response = restTemplate.exchange("/api/v1/resource", HttpMethod.POST, entity, MyResponse.class);
```

### 2. Rely on Dynamic IDs (Never Hardcode ID 1)
Database identity counters (`IDENTITY`/`SEQUENCE`) do not automatically reset between Cucumber scenarios. Hardcoding ID `1` in your requests or payloads will cause test flakiness or `StaleObjectStateException`.
*   **Action:** Capture the real ID generated by the database during the `Given` setup phase into a `static` variable and map any static reference from Gherkin (e.g., ID 1) to this dynamic variable at runtime.

## 🚀 Test Execution & Validation

Developers can trigger test cycles using the standard build automation tool in the WSL environment.

### Command Reference

*   **Standard Execution:**
    To run the entire test suite, compile the assets, spin up the ephemeral database containers, run Flyway migrations, and execute the test classes, use:
    ```bash
    mvn clean test
    ```

*   **Full Build & Coverage Generation:**
    To execute all verification phases and compile the final coverage reports via JaCoCo, run:
    ```bash
    mvn clean install
    ```

### Post-Execution Metrics
*   **Code Coverage Verification:** After completing a full build, navigate to the `target/site/jacoco/` folder and inspect `index.html` to review behavioral coverage criteria.
*   **Schema Consistency:** Verify that database migrations do not conflict with existing Hibernate definitions by reviewing the output trace for any SQL state warnings.
